# Hostlet 0.1.0 Release Review

Date: 2026-05-21 UTC

Reviewed release: `v0.1.0`

Release commit: `9f0879634563e7cec1763a9c06835a6bd358054e`

Release URL: https://github.com/ShaneKanterman04/Hostlet/releases/tag/v0.1.0

## Verdict

`0.1.0` is viable as a technical-user beta for a single-owner homelab deployment panel. The core build, web, API, agent, Compose, release, and self-hosted runner paths are working and CI is green on the `homelab` self-hosted runner.

It is not yet production-hardened. The main gaps are security controls around a tunnel-exposed control plane, durable job execution, setup simplicity, release supply-chain hardening, and automated security scanning.

## Release Inventory

- CLI release asset: `hostlet-linux-x64`, 6,014,400 bytes.
- Release checksum asset: `hostlet-linux-x64.sha256`.
- CLI format: Linux x86_64 ELF, dynamically linked against glibc system libraries.
- CI runner target: `[self-hosted, Linux, X64]`.
- Control-plane deployment path: cloned repository plus compiled CLI plus Docker Compose.
- Runtime services: PostgreSQL, Rust API, Next web UI, local Rust agent, Caddy, optional cloudflared.

## Strengths

- Production Compose binds API and web to loopback and routes public ingress through Caddy/cloudflared.
- First-run password, setup token, signed cookies, OAuth state validation, and CORS/origin checks are present.
- Secrets are generated by the CLI and `.env` is written with restrictive permissions.
- GitHub tokens and app env vars are encrypted at rest.
- Agent jobs are HMAC-signed before execution.
- Private GitHub repository deploys are supported through tokenized fetch URLs with log redaction.
- Deploys enforce a per-app active deployment lock.
- Webhook branch deletion and zero-SHA pushes are ignored.
- App deletion now requests agent cleanup of containers, images, and Caddy route state.
- Production Docker builds for web, API, and agent pass in CI.

## Findings and Fix Plan

### High: Public Control Plane Lacks Abuse Controls

Cloudflare Tunnel UI/API mode intentionally exposes the Hostlet UI, API, and webhooks through one public hostname. The current security model has owner allowlisting and CSRF/origin checks, but no API rate limiting, no audit log UI, and no role-based access control.

Evidence:

- `docs/SECURITY.md` lists missing audit log UI, RBAC, rate limiting, and scanning.
- `README.md` documents tunnel exposure for the UI/API.
- API routes include setup, unlock, GitHub Device Flow, deploy, rollback, server registration, and app mutation endpoints behind the same public control plane.

Strong fix:

- Add rate limiting middleware for `/api/setup`, `/api/unlock`, `/auth/github/device/start`, `/auth/github/device/poll`, `/webhooks/github`, app mutations, and deploy actions.
- Add an `audit_events` table and write audit events for login, unlock, app create/update/delete, deploy, rollback, tunnel open/close, env var changes, server registration, token rotation, and logout.
- Surface audit events in Settings and per-app views.
- Keep single-owner mode as default, but add a real owner/admin/viewer permission model before multi-user use.
- Consider documenting Cloudflare Access in front of the control plane as a recommended production layer.

### High: Agent Work Is Not Durable Enough

Deploy and rollback jobs are sent directly over the in-memory websocket agent map. Delete-app cleanup uses a database `agent_jobs` row, but finalization is performed by an API `tokio::spawn` task. If the API restarts while a deploy, rollback, or delete finalizer is in flight, records can remain stale or partially cleaned until manual recovery.

Evidence:

- `apps/api/src/deploy.rs` sends jobs through `state.agents` and fails immediately if the agent is offline.
- `apps/api/src/web.rs` spawns delete finalization in-process after sending the agent job.
- There is stale deployment recovery, but no equivalent startup reconciliation for unfinished `agent_jobs`.

Strong fix:

- Move deploy, rollback, delete, cleanup, stop, start, and restart onto one durable `agent_jobs` queue.
- Have agents poll or claim jobs from the API after reconnect instead of relying only on in-memory websocket delivery.
- Make every agent job idempotent with clear phases: queued, claimed, running, cleanup_pending, success, failed.
- Add startup reconciliation for running jobs and expired claims.
- Keep websocket as a low-latency notification channel, not the only job transport.

### High: Setup Is Better, But Still Not Simple Enough

The release now has a compiled CLI, but a first-time user still needs Docker/Compose, Git, curl, a repo clone, a GitHub OAuth app, and enough Cloudflare knowledge to answer the tunnel prompts. The README says to create the OAuth app when prompted, while the CLI asks for OAuth values during `hostlet init`, which can interrupt the flow.

Evidence:

- `README.md` requires Docker/Compose, Git/curl, and a GitHub OAuth App before setup.
- `apps/cli/src/main.rs` asks for the GitHub OAuth App Client ID during `hostlet init`.
- The compiled CLI still expects to run from the cloned repository root because Compose files, scripts, and Docker build contexts live in the repo.

Strong fix:

- Add `hostlet doctor --preflight` and run it before `hostlet up`.
- Add `hostlet install` for Linux that installs Docker/Compose if missing, validates architecture, and places the binary.
- Let `hostlet init` create a partial `.env`, validate that Device Flow is enabled when possible, and print the next UI login step.
- Add `hostlet init --non-interactive` for scripted installs.
- Package the control-plane Compose assets with the release, or publish a release tarball so users do not have to understand the repo layout.
- Add a one-page "fresh Ubuntu server" path that starts from zero and ends with a reachable UI.

### Medium: Release Artifact Is Narrow and Unsigned

The release binary is Linux x86_64 only and dynamically linked. It has a checksum, but no signature, provenance, SBOM, or documented checksum verification step.

Evidence:

- `.github/workflows/release.yml` builds only `hostlet-linux-x64`.
- `file target/release/hostlet` reports a dynamically linked Linux x86_64 ELF.
- `ldd target/release/hostlet` shows glibc system library dependencies.

Strong fix:

- Build `linux-x64-gnu`, `linux-x64-musl`, and `linux-arm64` release assets.
- Prefer a static musl binary for broad Linux server compatibility.
- Generate checksums for all assets and document `sha256sum -c`.
- Add Sigstore/cosign signing and GitHub artifact attestations.
- Generate a release SBOM for Rust and web dependencies.

### Medium: Security Scanning Is Not Enforced in CI

`pnpm audit` passed locally, but `cargo audit` found `RUSTSEC-2023-0071` for `rsa 0.9.10` through `sqlx-mysql` in `Cargo.lock`. `cargo tree` did not show it in the active Postgres target tree, so this appears to be inherited lockfile/dependency surface rather than active MySQL runtime use. It still needs explicit tracking. Container image scanning is not present.

Evidence:

- Local `cargo audit` result: `RUSTSEC-2023-0071`, `rsa 0.9.10`, no fixed upgrade available.
- `docs/SECURITY.md` says no image vulnerability scanning is integrated.
- `.github/workflows/ci.yml` runs build/test checks, but no dependency or image scanners.

Strong fix:

- Add `cargo audit` to CI with an explicit ignore file only if the advisory is documented as non-exploitable for Hostlet's active feature set.
- Add `pnpm audit --audit-level moderate`.
- Add `trivy` or `grype` scans for web/API/agent images.
- Fail CI on high/critical findings unless an expiry-dated exception is checked in.
- Revisit `sqlx` feature selection or lockfile contents to remove unused MySQL/RSA packages if possible.

### Medium: Remote Agent Install Is Deferred

Remote VPS support is intentionally out of scope for 0.1.0. The UI/API, Caddy, local agent, and deployed app containers run on the same host. Remote server creation, generated install commands, and remote agent registration now return a deferred/unavailable response.

Follow-up before re-enabling remote agents:

- Publish prebuilt `hostlet-agent` release binaries alongside the CLI.
- Issue per-server signing secrets and scoped tokens, with rotation and revoke paths.
- Make the install flow detect distro, architecture, systemd availability, Docker availability, and Caddy state before mutating the host.
- Add uninstall, reinstall, recovery, and API-side agent version/status display.
- Validate the full path against disposable VPS fixtures before marking it supported.

### Medium: Generated Static Node Deploys Are Fragile

For Vite, Astro, and SvelteKit without a Dockerfile or start script, Hostlet generates a Dockerfile that runs `npx serve -s dist` at container startup. That depends on runtime package download or an app-provided `serve` dependency, assumes the output directory is `dist`, and may fail in offline or restricted environments.

Evidence:

- `apps/agent/src/main.rs` maps static frameworks to `__hostlet_static`.
- The generated Dockerfile uses `CMD ["npx", "serve", "-s", "dist", ...]`.

Strong fix:

- Install `serve` during image build, or use a small static web server image.
- Detect and configure output directories per framework.
- Add SPA fallback control and static asset smoke tests.
- Add fixture deployments for Vite, Astro, SvelteKit, Next, simple Node, and Dockerfile apps in CI or a nightly workflow.

### Medium: Retention and Cleanup Are Incomplete

There is a cap of 20,000 stored log lines per deployment, but no global retention policy for deployment logs, failed containers preserved for inspection, old images, webhook events, agent jobs, or resource snapshots beyond delete-app cleanup.

Strong fix:

- Add retention settings for logs, images, failed containers, webhook events, and resource snapshots.
- Add an agent cleanup job that is safe and previewable.
- Show disk usage per server and per app.
- Add a default policy that keeps recent successful deployments and failed deployment evidence for a bounded period.

### Medium: Backup and Restore Need a Real Production Story

Backup/restore scripts exist, but docs still contain stale text saying backup/restore tooling is missing. The restore path is not validated in CI or with a clean-machine smoke test, and backup does not include live secrets by design, which is correct but easy for users to miss.

Evidence:

- `scripts/backup.sh` and `scripts/restore.sh` exist.
- `docs/SECURITY.md` still lists "no backup/restore tooling" as a highest-priority gap.

Strong fix:

- Update security docs and feature gap docs to match current state.
- Add `hostlet backup` and `hostlet restore` docs that clearly state `.env` must be stored separately.
- Add a restore smoke test procedure against a clean Compose project.
- Add scheduled/off-host backup guidance.

### Low: CI Has Near-Term Maintenance Warnings

CI is green, but GitHub reports Node.js 20 action deprecation warnings for common actions, and Docker build logs warn about the legacy builder.

Strong fix:

- Update actions as Node 24-compatible versions become available.
- Set `FORCE_JAVASCRIPT_ACTIONS_TO_NODE24=true` after validating the runner.
- Move Docker builds to `docker buildx build`.

### Low: Tunnel Wording Is Overloaded

The docs now clarify that "Cloudflare Tunnel for Hostlet UI/API" means the Hostlet control plane is reachable through the tunnel, while apps are private by default. In the UI, app actions still use "Open tunnel" and "Close tunnel", which can be confused with the control-plane tunnel mode.

Strong fix:

- Rename app actions to "Make public" and "Make private" or "Publish app URL" and "Unpublish app URL".
- Keep "Cloudflare Tunnel for Hostlet UI/API" only for control-plane access mode.
- In app settings, show the exact domain and whether DNS exists.

## Edge-Case Matrix

| Area | Current behavior | Risk | Fix |
| --- | --- | --- | --- |
| Fresh Linux install | Requires external Docker/Git/curl/OAuth knowledge | Setup friction | `hostlet install`, preflight doctor, partial init |
| No repo clone | CLI cannot operate because assets live in repo | Binary alone is not enough | Release tarball or embedded assets |
| Old Linux/glibc | Dynamic binary can fail | Install failure | Static musl asset |
| API restart during deploy | Stale deployment recovery exists after 30 minutes | Delayed or manual recovery | Durable agent job queue |
| API restart during delete | In-process finalizer can disappear | App DB row or DNS cleanup may remain | Durable finalizer with reconciliation |
| Agent offline | Deploy fails immediately | No queued retry | Durable queue and retry policy |
| GitHub token revoked | Repo listing/deploy fails | User confusion | Token status and reconnect flow per app |
| Branch delete webhook | Ignored | Good | Keep existing tests |
| Public app DNS opened before route | DNS may resolve to Hostlet Caddy 404 until deploy succeeds | Confusing public state | Tie public exposure to deployed route or show pending route |
| User Dockerfile runs as root | Trusted to define runtime user | App isolation risk | Policy warnings and optional hardening checks |
| Generated static app | Assumes `dist` and `npx serve` | Runtime failure | Build-time static server and framework fixtures |
| Log secrets | Best-effort redaction only | Secret disclosure | Continue redaction, document risk, add secret-pattern tests |
| Backups | DB and agent state backup exist; `.env` intentionally excluded | Irrecoverable encrypted data if `.env` lost | Strong docs and off-host secret backup guidance |

## Recommended Next Release Plan

### 0.1.1: Immediate Hardening and Clarity

- Update docs to remove stale backup/restore and settings gaps.
- Add checksum verification instructions.
- Add CI jobs for `cargo audit`, `pnpm audit`, and image scanning.
- Document the `rsa/sqlx-mysql` audit finding with an expiry-dated ignore only if accepted.
- Rename public app tunnel actions to reduce confusion.
- Add preflight checks before `hostlet up`.
- Add a fresh-machine setup walkthrough.

### 0.2.0: Reliability Backbone

- Implement durable `agent_jobs` for all deployment operations.
- Add agent claim/retry/reconciliation.
- Add audit event storage and UI.
- Add API rate limiting and unlock/OAuth backoff.
- Publish prebuilt agent binaries.
- Add release signing, attestations, and static Linux assets.

### Later Production Track

- Add RBAC and session management.
- Add scheduled off-host backups.
- Add retention and cleanup policies.
- Add remote server detail, token rotation, uninstall, and version reporting.
- Add broader runtime presets and deployment fixture tests.

## Final Assessment

Ship `0.1.0` only as a technical beta. It is good enough for controlled single-owner testing, especially on the known self-hosted runner and homelab environment. It should not be presented as production hardened until durable job execution, public control-plane abuse controls, security scanning, and release signing are in place.
